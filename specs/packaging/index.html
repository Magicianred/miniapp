This specification defines the standardized MiniApp packaging construction method, which provides a standard package structure description. The MiniApp package file packages all the static page templates / CSS / JavaScript files and the manifest file. The MiniApp package file can be used in any runtime environment (user agent).

本文描述了一个标准化的MiniApp包构建方式，提供一种标准的包结构描述，将小程序页面所需的所有静态页面模板、CSS 和 JavaScript 文件，和配置文件（Manifest）打包在一起，允许在任何运行时环境可使用。

The Miniapp package file contains a manifest document (located in the root directory of the package). The manifest file includes an overview of the MiniApp, including page description and page path. The package file contains JavaScript code, one or more page files, including template code of page(s), CSS code for page style and JavaScript code for page logic. The MiniApp package file should support digital signature verification.

小程序包资源，包含一个配置文档，位于包的根目录下。配置文件应包括整个小程序的概述（页面描述及其对应路径，用于页面定位和开启）。包含 JavaScript 代码的应用级逻辑文件，一个或多个页面文件，包含页面结构的模板代码、页面样式的 CSS 代码和页面逻辑的 JavaScript 代码。小程序包应支持数字签名验证。

The MiniApp runtime environment (user agent) can identify MiniApp package files through standard package format and MIME media type. After fetching the MiniApp package file, all static page templates / CSS / JavaScript files are cached on the device through a standardized dereference step. These cached package resources are available until the next update without fetching the MiniApp package file again. In terms of launch mode, the MiniApp can be launched normally without network. The runtime environment (user agent) locates the specified startup page from the package cache path and starts the MiniApp according to the description of the manifest file.

小程序运行时环境，可通过标准的文件格式、媒体类型识别小程序包文件。获取（Fetch）小程序包后，通过标准化的解引用，将所有静态页面模板、CSS 和 JavaScript 文件缓存在设备上。在下次更新之前，这些资源始终可用，无需重新获取（Fetch）小程序包文件。启动方式上，小程序无需网络也可以正常启动，运行时环境根据小程序标准配置文件的描述（manifest），从包缓存路径中定位指定的启动页面，启动小程序。

This is still a work in progress. The proposal is being incubated in the W3C MiniApps Ecosystem Community Group.

Packaging

Each MiniApp package is wrapped as a [[ZIP]] file. It contains, at the root level, a global configuration file (manifest.json) to describe and configure the whole MiniApp, an application logic file (app.js), and a default style file (app.css).

The MiniApp package also contains multiple page related files in the pages directory, and other common resource files in the common directory

One typical illustrative example of a MiniApp package is the following:

            |___manifest.json
            |___app.js
            |___app.css
            |___pages
            |       |___page1.js
            |       |___page1.xml
            |       |___page1.json
            |       |___page1.css
            |___common
            |       |___componentA.js
            |       |___componentA.xml
            |       |___componentA.css
            |       |___componentA.json
            |       |___example.png
            |___i18n
            |       |___zh-Hans.json
            |       |___en-US.json

root files

The MiniApp package contains at the root level the global data and lifecycle management of the MiniApp in the form of the following files:

  • manifest.json is responsible for the global configuration of the MiniApp, such as the file paths of MiniApp pages and the window configuration (e.g. navigator bar, background image and color, page title, etc.). Details are specified in the Manifest specification
  • app.css is responsible for the global CSS style for all MiniApp pages.
  • app.js is responsible for the service logic of the MiniApp as well as the lifecycle management of it, such as launching, showing and hiding the MiniApp.[[MiniApp Lifecycle]]

pages

The pages directory contains sets of files for the display and user interaction of all MiniApp pages. Each set of files that shares the same base file name (e.g. 'page1') with different extension names describe a particular page on different aspects, such as the service logic (e.g. page1.js), the configuration (e.g. page1.json), the structure (e.g. page1.xml) and the style (e.g page1.css). Developers can choose to put all page files directly under the pages directory in a flat manner, or organize them in different sub-directories for different pages.

  • A .json file is responsible for the configuration of a MiniApp page.
  • A .xml file is responsible for the structure of a MiniApp page, similar to HTML.
  • A .css file is responsible for the CSS style of a MiniApp page.
  • A .js file is responsible for the service logic and lifecycle management (defined in MiniApp Lifecycle) of a MiniApp page.

common

The common directory contains common resources such as components, multimedia resources, and utils (js files). Developers can choose to put all resource files directly under the common directory in a flat manner, or organize them in different sub-directories as needed.

i18n

The i18n directory contains multi-language configuration files to support internationalization. The name of each .json file in this directory follows the Language-Tag as defined in [[BCP47]], and represents the specific configuration for that particular language.

Note: The detailed format of the .json file is to be further developed.

Dereferencing

The essential steps for dereferencing a miniapp package contains 7 steps. User Agent SHOULD follow the rules to process the package archive file. And User Agent SHOULD execute failure process according to the package dereferencing algorithm.

Step 1 - Acquire a Miniapp Archive

User Agent MAY obtains a MiniApp archive and verifying MiniApp archive through specific channels(unspecified, e.g. network or offline). User Agent MAY fetch a potential MiniApp archive through data transfer protocol, such as HTTPs, WebSocket, FTP or specific TCP/UDP based protocols.

Verify Zip archive

The MiniApp package archive file is packaged as a specialized ZIP file. It can be delivered to the User Agent through different channels (unspecified, e.g. web, app store, offline), then installed/executed by the User Agent. User Agent NEED verify the ZIP archive through following algorithm.

1. Get a potential Zip archive.

2. Read the first 4 bytes of data in the potential Zip archive file.

3. Check if these 4 bytes are 50 4B 03 04.

4. If verified, process as vaild ZIP archive.

5. Otherwise, process as invaild ZIP archive.

Verify the MiniApp MIME Type

The User Agent RECOMMEND obtains potential ZIP archives from data transfer protocols, such as HTTPs. Regardless of whether its extension is .zip or no contain extension, User Agent NEED process the specific algorithm to check the potential archive.

If the obtained potential Zip archive is defined in an unsupported MIME Type, User Agent SHOULD process the potential Zip archive as an invalid MiniApp package file.

Step 2 - Verify the MiniApp Package File

There may be network errors and security issues during the fetching of MiniApp package archive. User Agent SHOULD verify the legitimacy of the archive. RECOMMEND to verify the digital signature file and integrity checking of archive. The dereferencing OPTIONAL process the decryption of binary files.

Integrity Check

If the MiniApp package is not data integrity or different summary. User Agent SHOULD regard the archive as an ineffective MiniApp package archive. And executes the failure proceess.

RECOMMEND User Agent to use the MD5 or SHA1 digest algorithm for verification. The Miniapp package digest algorithm is RECOMMEND using SHA-256 to ensure the security.

Encryption

The MiniApp package encryption algorithm RECOMMEND to use RSA encryption, and passes the key verification to prevent the MiniApp package from being tampered during network transmission. For the encryption algorithm key length, is RECOMMEND to use RSA 4096 bit length.

Step 3 - Locate and Process the Digital Signature

Miniapp Digital Signature

User Agent SHOULD locate the digital signatures in archive. The digital signatures SHOULD packaging in archive file with other resources together. User Agent SHOULD process the failure execution while digital signatures checking failed.

Locate Signature File

User Agent SHOULD verify the package digital signature file and check the digital signature identity before installing the MiniApp package. If the digital signature verification failed, it should be regarded as an ineffective package archive. User Agent SHOULD execute the MiniApp package installation failure process.

The User Agent SHOULD locate the X.509 Certificate file [[RFC5280]] in the MiniApp package archive. It is RECOMMEND using an algorithm with sufficient strength, key length, and certificate format. The MiniApp digital signature checking follows the [[RFC5280]] specification.

Verify Digital Signature

RECOMMEND using X.509 version 3 [[RFC5280]] Spec digital certificates to build a secure channel for fetching package archive to prevent man-in-the-middle attacks.If the signature file is empty (this means that the signature file was not found in the MiniApp package archive) , the signature is verified to incorrect. User Agent SHOULD execute the process of the installation failure.

Step 4 - Locate the Manifest File

Locate Manifest File

User Agent SHOULD executes an failure installation process while the manifest.json file does not exist or the parsing process is wrong.

User Agent SHOULD use JSON parser to process manifest.json, and call the parsed object a manifest.

Step 5 - Process the Manifest File

Process Manifest File

User Agent SHOULD parse the attributes in the manifest.json file[[MiniApp Manifest]], which contains the MiniApp’s meta data. The meta data is the attribute values used for MiniApp runtime.

Step 6 - Process Runtime

Process Platform Runtime

User Agent NEED to prepare the runtime environment. It contains JavaScript Framework to locate the first page in the MiniApp package.

User Agent runtime is isolated with two parts, the logical container and the rendering container. The logical container loads logical code (such as app.js, page.js files). The rendering container locate and loads the page resource files (such as template HTML and CSS files).

Note: Recommending User Agent JavaScript V8 engine as the logic container. Rendering container MAY tilt toward Webview or other rendering containers depend on specific User Agent.

The Package Reload

User Agent NEED fetch the orign package file at first time installation. And User Agent SHOULD cache the Max-Age of specific version of package archive. User Agent MAY Only load business data without reFetch the package archive again while MiniApp Reload.

Step 7 - Locate the Start File

Locate the Start File in Logical

The logic container start file of Mini App is app.js, which is responsible for life cycle management, launching, showing and hiding. User Agent SHOULD find the logical start file app.js of MiniApp.

Locate the Start File to Render

Each MiniApp SHOULD have a start file, which is responsible for the structure, similar to HTML.

User Agent SHOULD locate the first page as which path MiniApp URI Scheme [[MiniApp URI Scheme]] defined while starting up MiniApp through the MiniApp URI.

If MiniApp URI Scheme's specific the path is empty, User Agent SHOULD use the routing configuration pages in manifest.json (pages should be the path of all pages in the MiniApp). The first page SHOULD be the default first page.

Security & Privacy Considerations

Integrity & Trustworthiness

To ensure the integrity and trustworthiness, a MiniApp package should be protected by one or more digital signatures by the author (e.g. the MiniApp developer) and/or distributors (e.g. an application store) along with certificates issued by trusted authorities.

  • A digital signature (with a valid certificate) by the author ensures the origin of the MiniApp, so that an end user or a hosting platform can decide whether to install the MiniApp package according to the knowledge about the author (e.g. credits, blacklist, quality).
  • A digital signature (with a valid certificate) by a distributor ensures the integrity of the package and trustworthiness of the delivery channel, so that the end user can be protected from tampered software and can benefit from a healthier ecosystem.

    • Proven technologies such as [[RFC5652]](i.e. PKCS#7) can be used as the solution of the digital signatures for MiniApp package. Further evaluation is expected regarding whether it needs to be standardized in detail (e.g. the content scope under protection, additional attributes of concern, file format of the signature block, procedures), or is left to the discretion of implementations.

    Confidentiality

    There is no requirement to develop a standardized encryption mechanism for the MiniApp package to protect its confidentiality. However, it doesn't preclude an implementation from deploying some encryption mechanism for special purpose.

    IANA Considerations

    The proposed MIME type for the MiniApp package is application/miniapp-pkg+zip. It needs to be registered with IANA when this work moves to a normative phase.

    Note: A temporary solution could be application/x-w3c-miniapp-pkg+zip for the sake of any early implementation.
